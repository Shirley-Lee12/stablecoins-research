# 一、统一设计:三个 facet 都受控、都可后续增补

`tags` 表用 `facet` 区分三类标签,`status` 区分正式/候选:
- `facet ∈ {theme, jurisdiction, asset}`
- `status ∈ {active, candidate}`
- 种子数据一律 `status=active`;打标时映射不进任何 active 标签的新词 → `status=candidate` 入候选队列,**三个 facet 都走这套**(不只是主题)
- 新币种、新辖区将来都通过"候选队列 → 管理员提升为 active"来增补,**不需要改代码、不需要重新部署**

# 五、对 U.3 的设计补充:打标必须可对全库重跑

> 原 U.3 把"打标"放在上传流程里。这里追加硬性要求:**打标逻辑必须设计为可对全库重跑的独立任务,为后续基于真实数据重构标签预留接口。**

- 打标逻辑抽成独立函数/服务 `retagResources(resourceIds?)`:不传参 = 全库重跑,传 id 列表 = 指定重跑
- 上传流程只是调用它处理单条;它**不得**与上传流程耦合死、只能跑一次
- 暴露一个管理员触发入口(后台按钮或脚本命令):"用当前标签词表,对全部资源重新打标"
- 重跑逻辑:对每条资源,用摘要与各 active 主题标签 definition 做 embedding 相似度匹配(最多 4 个主题);币种/辖区做受控实体匹配;映射不进的进候选队列
- 重跑是幂等的:先清掉该资源旧的 theme/asset/jurisdiction 关联(保留人工手动加的标记,见下),再按新词表重建
- **保护人工修正**:`resource_tags` 加一个 `source` 字段(`auto` / `manual`),重跑只覆盖 `auto`,不动管理员手工加的 `manual` 关联
- 验收:改动词表后,跑一次重打标,全库标签按新词表刷新,人工标记不丢

# 六、后续 commit(灌完 100+ 篇后再做):标签重构辅助面板

> 目的:攒够真实数据后,用数据驱动地决定标签怎么拆/砍/增。**工具只负责"发现"与"执行",增删标签的决策由管理员做。**

管理后台新增"标签分析"页,三块:

1. **覆盖度统计**:每个 active 标签挂了多少篇资源(分 facet)。一眼看出哪些过载(该拆)、哪些荒芜(该砍)。纯聚合查询。
2. **候选队列审核**:列出所有 `status=candidate` 的标签及其关联文献数,管理员可"提升为 active / 合并到已有标签 / 删除"。
3. **聚类发现**:对全库摘要做 embedding 聚类,可视化呈现自然聚成几类、各类主题词;与现有 active 标签对照,提示"某聚类横跨多个标签或不属于任何标签"——作为调整词表的信号。

操作闭环:**面板发现 → 管理员决策(拆/砍/增/合并)→ 改词表 → 触发第五部分的全库重打标**。

- 验收:能看到每标签文献数、能审核候选标签、能看到聚类结果;改词表后重打标全库刷新
