# 第一组补充:状态机修正(重要,影响第二组设计;本节覆盖12号文件的旧规则)

## 0.1 核心原则

- **只有`approved`(管理员审核通过)的资源,才会出现在网站公开的Resources界面**
- 其余所有状态的资源,**只在提交用户自己的界面(My Contributions)可见**,不出现在公开列表,也不占用管理员的审核视野(除了`pending`)
- 管理员是**最后一道把关**,只处理"系统判断不了、需要人的价值判断"的情况

## 0.2 判断标准:什么问题该"打回给用户",什么问题该"留给管理员"

**分界线:用户拿到这个反馈后,有没有一个具体能做的动作去修正它。**
- 能修正 → 系统自动打回给用户,不占用管理员时间
- 不能修正(需要编辑判断,不是用户改一改就能解决的)→ 留给管理员做最终取舍

按这条标准,四类问题符合"打回给用户":

| 状态 | 中文显示 | 触发条件 | 用户能做的修正动作 |
|---|---|---|---|
| `incomplete` | 待补充 | 六要素(标题/作者/年份/摘要/关键词/URL·DOI)任一缺失 | 补充信息 |
| `disputed` | 待核实 | 六要素齐全,但DOI/URL解析结果、年份、作者与权威记录不一致 | 核实并修正错误字段 |
| `off_topic` | 与稳定币无关 | 打标后主题标签(theme facet)完全为空(见0.4) | 确认相关性或主动撤回 |
| `duplicate` | 疑似重复 | DOI/URL与库中已有资源精确匹配,或标题高度相似(见0.5) | 确认是否重复,撤回或说明是不同版本 |

两类问题**不满足"用户能修正"这条标准,继续留给管理员**做最终取舍:
- **来源质量存疑**(如疑似掠夺性期刊):期刊可信度不是用户能改变的,这是编辑判断
- **信息真实性存疑**(如疑似AI编造/引用造假):需要经验判断,用户自证不了

## 0.3 管理员最终拒绝理由(精简后,3个)

12号文件原有9个拒绝理由,这次连同上一轮已经拆出去的几类一起,最终收窄为:**来源质量存疑、信息真实性存疑、其他**(3个)。

**Schema动作**:更新`rejection_reasons`表的种子数据,只保留`low_quality_source`/`authenticity_concern`/`other`这3条,移除其余6条(它们现在都对应`incomplete`/`disputed`/`off_topic`/`duplicate`四种状态的自动判定原因,不再是管理员手动选择的理由)。已有测试数据里引用了被移除理由的历史记录,不影响新数据,可以忽略。

## 0.4 与稳定币研究方向是否相关(off_topic)

- 现有核对agent(verify)核对的是"字段对不对得上",不判断"内容主题相关性",需要新增这一层判断
- 建议:复用打标签的结果——**如果一篇资源被`retagResources()`打标后,主题标签(theme facet)完全为空(一个都没命中)**,判定为`off_topic`,备注写"未能识别出相关研究主题,请确认是否与稳定币研究相关"
- 这是一个辅助信号,不追求100%准确,用户可以在My Contributions里申诉/确认相关性后重新提交

## 0.5 重复资源检测(duplicate)

- **精确匹配(强信号,直接判定)**:提交的DOI或URL,与库中已有的任意一条资源(不限状态)完全一致
- **标题模糊匹配(弱信号,仅提示确认,不直接判死)**:标题去除标点空格后高度相似 + 年份相近,判定为"疑似重复",提示用户确认
- **⚠️ 已知的合法例外,不要误判**:同一份研究可能存在多种不同形态(如SSRN工作论文版 + 正式出版的书,DOI/URL不同但标题相近)——这种情况不应被精确匹配命中(因为DOI/URL本来就不同),标题模糊匹配命中时,用户确认"这是不同版本/不同出处"后应该能正常放行,不强制要求撤回

## 0.6 多个问题同时命中时,判定优先级

一条资源可能同时触发多个条件(比如既缺摘要、DOI又对不上)。**只取最优先的一个作为当前状态**,按以下顺序判断(前面的优先):

`duplicate` > `incomplete` > `off_topic` > `disputed`

理由:重复资源最优先判断(如果确实是重复,后面的检查都没有意义);完整性其次(信息不全,后面的核对/主题判断意义不大);主题相关性再次;最后才是字段核对。**但详情页展示时,应该把所有检测到的问题都列出来给用户看**(不只是显示优先级最高的那一个),避免用户改完一个问题重新提交,又冒出下一个没被提前告知的问题,增加不必要的往返。

## 0.7 用户操作流程

1. 用户在My Contributions看到打回状态的条目(四类之一),清楚展示具体原因(可能不止一条,见0.6)
2. 用户**主动修改**对应内容,或者对`off_topic`/`duplicate`选择"确认相关/确认非重复"这类申诉动作,重新提交
3. **系统重新跑一次完整检查**(完整性 + 核对agent + 主题相关性 + 重复检测),全部通过才真正进入`pending`
4. `pending`:用户界面标注"待审核";同时出现在管理员审核队列
5. 管理员审核:通过 → `approved`,进入公开Resources界面;拒绝 → `rejected`(3个精简理由之一),用户可修改后重新提交回到对应打回状态重新走流程

## 0.8 覆盖12号文件的旧规则

- **12号文件规定"手填/URL入口,硬性必填缺失时服务器直接拒绝确认(400,不创建记录)"——本次废止**
- 新规则:任何入口,六要素缺失都不再拒绝提交,而是正常创建记录,状态设为`incomplete`,用户已录入内容被保留
- 请移除`persistConfirmedDraft()`里针对手填/URL入口的硬性拒绝逻辑(`MissingRequiredFieldsError`部分)

## 0.9 状态机总览(最终版,七个状态)

| 状态 | 中文显示 | 可见范围 |
|---|---|---|
| `incomplete` | 待补充 | 仅用户自己 |
| `disputed` | 待核实 | 仅用户自己 |
| `off_topic` | 与稳定币无关 | 仅用户自己 |
| `duplicate` | 疑似重复 | 仅用户自己 |
| `pending` | 待审核 | 用户自己 + 管理员审核队列 |
| `approved` | 已通过 | **全站公开(唯一会出现在Resources页面的状态)** |
| `rejected` | 已拒绝 | 用户自己(附拒绝理由,可修改重提) |

`uploading`/`processing`仍然只属于`upload_jobs`表,不是`resources.status`的值。

## 0.10 待你确认

- **"作者信息有误"**:已放进"待核实"(系统能交叉比对权威记录),如果你认为作者核验目前不够可靠,可以改放进管理员最终理由里,告诉我调整
- 0.2的"用户能否修正"这条判断标准,以后遇到新的边界情况,可以按这个框架自己判断该归到哪一类,不需要每次都回来确认

## 1.1(对应反馈#6)审核队列查询条件确认

- 管理员审核队列(Approvals)只需要显示`status='pending'`——`incomplete`/`disputed`/`off_topic`/`duplicate`四类打回状态都在更早的阶段被系统拦下,不会到这一步

- 验收:点击拒绝必须先选理由才能提交,后端`action='reject'`时`rejectionReasonId`为空应返回400

# 第二组:My Contributions 页面 + 审核流程重建

> 这一组解决反馈#1、#3、#7(除1.2以外的部分),它们本质是同一套东西——用户需要一个地方追踪"我提交的东西现在怎么样了",管理员需要一套完整的"先看详情、再决定"的审核流程。

## 2.1 My Contributions 页面(从"Coming Soon"占位改为真实功能)

- 展示当前用户提交过的**全部**资源,覆盖所有状态,不只是"已批准"的:
  - `uploading`/`processing`(仍在`upload_jobs`里处理中的,还没成为正式resources行的)
  - `incomplete`(待补充)
  - `disputed`(待核实)
  - `off_topic`(与稳定币无关)
  - `duplicate`(疑似重复)
  - `pending`(待审核)
  - `rejected`(已拒绝,连同拒绝理由展示给用户)
  - `approved`(已通过)
- **页面顶部要有一个汇总统计条**(对应反馈#3提到的"dashboard"):比如"共12条 · 处理中2 · 待补充1 · 待核实1 · 待审核3 · 已拒绝2 · 已通过4"这样的数字概览(打回类的四种可以合并展示成一个"需处理"总数,点开再看细分)
- 每条资源可点击查看详情(遵循1.3的统一跳转规则)
- 需要同时查询`upload_jobs`(处理中的)和`resources`(已经落库、有状态的)两张表,合并展示成一个统一列表
- 这个页面天然解决反馈#1:用户提交后可以直接关闭上传弹窗,处理仍在后台(job)继续进行,用户随时回**My Contributions**查看进度,不需要一直守在上传弹窗前等待

## 2.2 管理员审核流程重建(对应反馈#7b/7c)

**当前问题**:审核队列里点击资源,直接就是批准/拒绝按钮,看不到资源完整信息(标题/作者/年份/摘要/标签/核对报告),不符合"先看信息再决策"的基本审核逻辑。

**要求的正确流程**:
1. 管理员在审核队列点击一条资源 → 打开**完整资源详情视图**(复用现有的资源详情组件/弹窗,不要重新做一套),展示六要素 + 核对报告
2. **说明**:由于四类打回状态的资源都已经被拦在管理员队列之外,管理员在这里看到的核对报告理论上应该都是全绿(✅)的——这是预期行为,不是bug;展示报告的意义是让管理员在批准前仍能一眼确认信息完整、无疑点,而不是要求管理员去处理疑点
3. 详情视图里提供"通过"/"拒绝"两个操作入口
4. 点"拒绝"→ 弹出受控理由下拉框(0.3精简后的3个理由:来源质量存疑、信息真实性存疑、其他)+ 可选文本框,选完理由才能真正提交拒绝(接1.2的修复)
5. 提交后返回队列,该条从待审核列表消失

## 2.3 审核记录/日志视图(对应反馈#7中"多维表格"的要求)

- **不需要新建一张独立的审计日志表**——12号文件已经在`resources`表加了`reviewedBy`/`reviewedAt`/`rejectionReasonId`/`rejectionNote`这几个字段,一条资源对应一次审核结果,目前项目里没有"同一条资源被反复审核多次"的重新提交流程,所以这几个字段已经够用,不必过度设计
- 新增一个管理员可见的"审核记录"列表视图,查询条件是`reviewedBy IS NOT NULL`(即已经被处理过的资源),展示字段:
  - **资源标题**(点击跳转站内详情页,遵循1.3规则)
  - **提交者邮箱**(用`resources.createdBy`关联到`users.email`,用邮箱作为唯一标识,不用显示名——这是对的判断,邮箱在系统里是唯一的,显示名不一定是)
  - **提交时间**(`resources.createdAt`)
  - **处理时间**(`reviewedAt`)
  - **处理人**(`reviewedBy`关联`users`表)
  - **处理结果**(`approved`/`rejected`,拒绝的话显示理由)
- 支持按状态/处理人/时间范围筛选
- 如果将来需要支持"资源被拒绝后允许用户修改重新提交、需要看多轮审核历史"这类功能,那时候再考虑升级成独立的审核日志表(每次操作一条记录),现在不需要

## 2.4 管理员可编辑资源信息(不只是通过/拒绝)

- 管理员在审核详情视图(2.2)里,除了"通过"/"拒绝",增加**直接编辑资源字段**的能力(标题、作者、年份、摘要、关键词、URL/DOI、标签),不需要逼用户重新提交一遍才能修正小问题
- **管理员编辑后不需要重新跑系统核对**——管理员的判断本身就是权威的,不同于用户自己修改后必须重新核对(0.7的规则只针对用户,不适用管理员)
- **需要留痕**:被管理员编辑过的字段,记录"该字段已被管理员修改"(可以是`resources`表加一个`adminEdited`布尔字段做粗粒度标记,或者更细的按字段记录,先做粗粒度的够用),方便以后追溯这条资源的信息是系统原始抽取的还是管理员手动改过的
- 编辑标签时复用现有的标签选择组件(受控词表 + 自由候选流程不变,管理员改标签走的是"人工标记",应该正确写入`resource_tags.source='manual'`,和T.4的人工保护机制衔接上,以后`retagResources()`重跑不会覆盖管理员手动改过的标签)

# 第三组:标签层级结构(对应反馈#4)

## 3.1 背景

T.2设计主题标签时,33个标签本来就分了六大类(A-F),但这个分类只存在于设计文档里,从未成为数据库里的字段。现在要做"折叠目录"效果,需要先把这层结构补进schema。

## 3.2 Schema改动

`tags`表新增字段:`category`(text,可空,**仅theme facet使用**,jurisdiction/asset的这个字段留空)

## 3.3 六大类 slug 及归属标签(用于种子数据回填)

| category slug | 中文名 | 归属的 theme 标签 slug |
|---|---|---|
| `types_mechanisms` | 类型与机制 | fiat-collateralized, crypto-collateralized, algorithmic, commodity-rwa-backed, cbdc, tokenized-deposits-mmf |
| `stability_risk` | 稳定性与风险 | peg-stability-depeg, run-liquidity-risk, reserve-quality-transparency, collateral-risk, smart-contract-security, custody-counterparty, systemic-contagion |
| `regulation_policy` | 监管与政策 | regulatory-frameworks, licensing-supervision, aml-cft, consumer-protection, disclosure-accounting, cross-border-coordination |
| `monetary_macro` | 货币与宏观 | monetary-transmission, dollarization-substitution, bank-disintermediation, capital-flows-sovereignty |
| `markets_adoption` | 市场与应用 | payments-remittances, defi-lending, trading-market-structure, adoption-emerging-markets, market-data-supply |
| `tech_infrastructure` | 技术与基础设施 | blockchain-chains, interoperability-bridges, oracles-data-feeds, privacy-compliance-tech, programmability |

请写一个迁移脚本,按上表把33个theme标签的`category`字段回填好。

## 3.4 前端:折叠树形展示

- 资源列表侧边栏的标签筛选区域,`theme` facet 改成两级折叠结构:一级是六大类(加粗、字号加大,比现有标签更突出),点击展开/折叠显示该类下的具体标签
- `jurisdiction`(可以按之前设计的region分组:Americas/Europe/APAC等)、`asset`两个facet的展示方式不强制改成树形,除非做起来顺手,不是本次重点
- 默认折叠状态:六大类默认全部折叠,用户点开哪个看哪个,避免侧边栏一开始就铺满几十个标签

## 3.5 打标相似度改为"标题+摘要加权",并把分数持久化(对应反馈#5的资源列表精简需求)

**背景**:资源列表页只显示一个"最核心的一级标签"(见6.1),需要知道"这条资源命中的几个主题标签里,哪个相似度分数最高"才能选出这一个。当前`retagResources()`的相似度计算,需要先确认现状:

- **请先确认**:当前打标逻辑是只用摘要算相似度,还是已经把标题也纳入了计算。如果只用了摘要,需要补上标题这部分。
- **改造后的逻辑**:标题单独和每个主题标签的定义句算一次相似度,摘要单独算一次,两者加权合并成最终分数(**标题权重更高**,建议初始比例标题60% : 摘要40%,写成可调的常量,不要写死在多处,方便以后统一调整)
- **理由**:文章标题通常代表核心意图,即使正文提到多个方面,标题所指向的方向应该在排序时占更大权重
- **这个权重比例是初始猜测值,不追求一次调准**——建议库里积累一定数量真实打标结果后,人工抽样核对"排出来的最核心标签,符不符合直觉判断",不对再调整比例,这轮不用做到完美

**Schema改动**:`resource_tags`表加一个`score`字段(数值类型,存最终加权后的相似度分数),之前这个数字在计算时产生但从未保存,现在需要存下来供列表页排序使用。

## 3.6 资源列表页"最核心一级标签"的选取逻辑

- 一条资源可能命中多个主题标签,可能分属不同的一级大类
- 取该资源命中的所有主题标签里`score`最高的那一个,用它所属的`category`(3.2的字段)作为列表页展示的"一级标签"
- **已知的信息损失,记录在案**:如果一篇资源确实横跨多个差异较大的领域(比如同时深入讨论了风险和监管),只显示一个一级标签会丢失另一个维度的信息,这是"列表页精简"和"信息完整性"之间的取舍,接受这个取舍,不在列表页做多标签展示;完整的标签信息在详情页(6.2)展示

# 第五组:资源关键词字段(对应反馈#9)

## 5.1 背景

题录导入时的`K1`字段之前只被用作**打标签的辅助输入**(帮助embedding相似度匹配),从未作为独立字段存下来给用户看。用户反馈:大部分文献本身自带"关键词"(和摘要一起出现在原文里),资源条目里应该展示出来;原文确实没有的,允许用LLM提炼补充,但要标注是AI生成的。

## 5.2 Schema改动

`resources`表新增两个字段:
- `keywords`(text[],可空,自由文本,不是受控词表——`tags`是受控分类体系,`keywords`是文献本身的关键词,两者是不同的东西)
- `keywordsSource`(text,标注来源:`extracted`原文提取 / `generated`AI生成 / `manual`用户手填)

## 5.3 各入口怎么填充这个字段(与摘要缺失时的处理逻辑保持一致)

**优先级顺序:原文提取 > 用户手填 > AI生成(需标注)**

- **题录导入(CNKI)**:直接取`K1`字段的值,按分号拆分存入,`keywordsSource='extracted'`(这个字段本来就在解析流程里,只是之前没有持久化,现在补上落库这一步)
- **PDF/URL 抽取**:LLM抽取时,优先查找原文里明确的"关键词:"/"Keywords:"章节并提取,`keywordsSource='extracted'`
- **手填**:提供可选的"关键词"输入框,用户自己填,`keywordsSource='manual'`
- **原文确实没有关键词章节、用户也没手填时**:允许(不是强制)用LLM从摘要/正文提炼关键词,**限制在3-5个**(prompt里明确约束数量),`keywordsSource='generated'`
- **完整性判定**:六要素里"关键词"这一项,只要`keywords`字段非空即算满足(不管是extracted/manual/generated哪个来源),不强制要求必须是"原文自带的"才算数——用户选择用AI生成来补全,也应该算作已完成(否则会一直卡在`incomplete`状态出不去)

## 5.4 前端展示

- 资源详情页/卡片上,`keywords`作为独立展示区块,和`tags`(受控标签)明确区分(不同标题,比如"关键词" vs "标签分类")
- 如果`keywordsSource='generated'`,在关键词旁边加一个明显的小标注(如"AI生成"角标或提示图标),不能和`extracted`/`manual`来源的内容在视觉上混同——这是诚信要求,不是可选项
- 摘要字段如果以后也走同样的"AI生成兜底"逻辑,同样需要标注来源,建议摘要也补一个`abstractSource`字段做统一处理(不在本次范围内,但可以顺手一起做,复用同一套设计)
