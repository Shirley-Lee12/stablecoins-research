## 一、URL/DOI 必填规则:按入口拆分,不是全局一刀切

**手填卡片、DOI/URL 导入(单条+批量)**——URL 或 DOI **升级为硬性必填**,和标题/作者/年份同一档。缺失时:
- 手填卡片:前端表单校验拦截,不允许提交
- DOI/URL 导入:本身以 URL/DOI 为输入起点,天然满足,不需要额外处理

**PDF 上传、题录批量导入**——URL/DOI **维持现状,不升级为硬性必填**。反查(U.2)找不到规范链接时,仍然走 `needs_review`,**不拒绝提交**。原因:这两个入口的链接是系统自动反查得到的,不是用户手填,反查注定有查不到的情况(政府法规网站403拦截、可疑来源、纯工作论文无公开记录等,均为已知客观限制,见10号文件)。把"自动化找不到"升级为"拒绝提交",不会减少人工负担,只会把负担提前到用户上传的当下,体验更差。

**需要新增的体验改进**:PDF/批量入口走 `needs_review` 时,若原因是"缺少 URL/DOI",请在核对报告和管理员队列里**明确标注**这一具体缺项(而不是笼统的"needs_review"),方便后续针对性批量补链接。

### 实现要点
- `determineResourceStatus()` 或对应的硬性必填校验逻辑,需要能区分"入口类型"(manual/url/pdf/batch),对 manual 和 url 入口额外校验 URL/DOI 非空
- 前端手填表单(`academic-resources.tsx` 手填 tab)加 URL/DOI 必填校验,和标题/年份用同一套提示逻辑
- PDF/批量入口的 `needs_review` 原因需要结构化(不是一段自由文本,而是能标识"缺失字段列表",至少包含 `missing: ['url_doi']` 这类可读取的结构),供审核队列展示

## 二、管理员审核:通过 / 拒绝 + 受控拒绝理由

### 2.1 审核操作
管理员对 `pending` / `needs_review` 状态的资源,可执行两个动作:
- **通过**:`status → approved`
- **拒绝**:`status → rejected`,必须选择一个拒绝理由(见下),可附加自由文本补充说明

### 2.2 拒绝理由:受控列表(单选主理由 + 可选自由文本补充)

新增一张小的受控列表(和 tags/sourceType 一致的 slug + 双语名模式),种子如下:

| slug | nameZh | nameEn |
|---|---|---|
| invalid_url_doi | URL/DOI 失效或不符 | Invalid or mismatched URL/DOI |
| author_error | 作者信息有误 | Incorrect author information |
| year_error | 年份错误 | Incorrect publication year |
| low_quality_source | 来源质量存疑 | Questionable source quality |
| off_topic | 与稳定币研究方向无关 | Not related to stablecoin research |
| duplicate | 重复资源 | Duplicate resource |
| incomplete_info | 摘要/关键信息不完整 | Incomplete abstract or key information |
| authenticity_concern | 信息真实性存疑 | Authenticity concern |
| other | 其他(见补充说明) | Other (see notes) |

### 2.3 Schema 改动
- 新增 `rejection_reasons` 表:`id, slug(unique), nameZh, nameEn`,种子插入以上 9 条
- `resources` 表加:
  - `rejectionReasonId`(可空,FK → rejection_reasons.id)
  - `rejectionNote`(可空 text,自由文本补充说明)
  - `reviewedBy`(可空,FK → users.id,记录是哪个管理员操作的)
  - `reviewedAt`(可空 timestamp)

### 2.4 数据处理原则
- **拒绝不等于删除**——`rejected` 状态的资源保留在库里,原上传者应能看到拒绝理由和补充说明,为将来"修改后重新提交"流程留空间(本轮不需要实现重新提交,只需保证数据和理由都保留、可查询)
- 拒绝理由是单选主理由,不做多选(保持统计口径干净,以后可以直接跑"拒绝原因分布")

### 2.5 后端路由
- `PATCH /api/admin/resources/:id/review`:body 包含 `{ action: 'approve' | 'reject', rejectionReasonId?, rejectionNote? }`,`action='reject'` 时 `rejectionReasonId` 必填
- 需要 `requireAuth, requireAdmin`

### 2.6 前端(管理员审核队列)
- 待审核列表(`pending`/`needs_review`)每条给"通过"/"拒绝"两个操作
- 点"拒绝"弹出下拉框(受控列表,单选)+ 可选文本框(补充说明)
- `needs_review` 的资源,若核对报告里标了"缺失 URL/DOI"(一、里提到的结构化缺失标注),队列列表要有视觉提示,方便管理员筛选出"专门需要补链接"的一批
