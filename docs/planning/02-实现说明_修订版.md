# 第二部分:注册邮箱验证 + 忘记密码(认证流程)

> 现状:`auth.ts` 的注册是「填邮箱+密码直接建号」,无验证码;忘记密码只生成 token 没发信。两处都要改。

## Commit A.1 — 注册邮箱验证码(新增)
新流程:**发码 → 校验码 → 设密码 → 建号**。
- DB 新增 `email_verification_codes` 表:`id, email, codeHash, expiresAt, consumed, createdAt`(验证码哈希存,5–10 分钟过期)
- 后端新增/改造路由:
  - `POST /auth/register/send-code`:输入 email → 校验邮箱未被注册 → 生成 6 位验证码 → 哈希入库 + `sendMail` 发到邮箱 → 返回"已发送"(做发送频率限制,防刷)
  - `POST /auth/register/verify`:输入 email + code + password + confirmPassword → 校验码未过期未用、两次密码一致 → 通过则建号 + 标记 code consumed + 返回登录 token
- 现有 `POST /auth/register` 改为上面两步,或保留为内部最后一步
- 验收:不收验证码无法注册;验证码过期/错误被拒;两次密码不一致被拒

## Commit A.2 — 前端注册流程
- 注册弹窗(`components/auth-dialog.tsx`)改两步:① 填邮箱→点"发送验证码";② 填验证码 + 密码 + 确认密码→提交
- 有验证码倒计时/重发
- 验收:UI 走通两步注册

## Commit A.3 — 忘记密码邮件重置(补全发信)
- `POST /auth/forgot-password`:已生成 reset token,这里接 `sendMail` 把**重置链接**(含 token,如 `/reset-password?token=...`)发到邮箱
- `/reset-password` 页(已存在)凭 token 让用户设新密码 → 校验两次一致、token 未过期未用 → 更新密码 + 作废 token
- 验收:点忘记密码 → 邮箱收到带链接的邮件 → 打开能重置成功

# 第四部分:上传管线(三入口 → 一条共用流水线)

## 终点统一:六要素 = 标题 + 作者 + 年份 + 摘要 + 标签 + 直达URL/DOI
> 平台**不存原文**,只存信息卡片与访问链接。

## 三入口前处理
- **手填卡片**:用户自填 → 直接进"核对"步
- **PDF 上传**:LLM 抽六要素 → 无 DOI/URL 时 agent 反查链接 → 进流水线
- **DOI/URL 上传**:agent 抓取/查库反查整篇信息 → 进流水线

## LLM vs Agent 分工
| 步骤 | 用什么 |
|---|---|
| 从 PDF/网页文本抽六要素 | LLM 单次 |
| 反查直达链接(标题→DOI) | Agent + 工具(联网,LLM 单次会编造) |
| 入库前核对六要素 | Agent + 工具(联网验证真实性) |

## Commit U.1 — agent 工具函数(`artifacts/api-server/src/lib/scholar/`)
- `searchCrossref(title)` — 正式发表论文
- `searchOpenAlex(title)` — 覆盖更广,返回作者+机构ID(喂 authors/institutions)+ OA 链接
- `searchSemanticScholar(title)` — 工作论文/预印本,返回 openAccessPdf
- `resolveDoi(doi)` — content negotiation 验证存在并取元数据
- `unpaywall(doi)` — 付费墙时找合法 OA 副本
- 验收:每个函数独立返回标准化结果

## Commit U.2 — 反查链接 agent
- 输入:LLM 抽出的标题/作者/年份
- 多源瀑布(Crossref→OpenAlex→Semantic Scholar);匹配判定 = 标题词重合 + 作者姓氏重合 + 年份接近,三者交叉够吻合才接受,防张冠李戴;都查不到返回 `not_found`
- 区分 `canonical_url`(规范落地页/DOI)与 `fulltext_url`(可打开的 OA 全文),另存 `access_status`
- 验收:无 DOI 论文能查到并交叉验证通过

## Commit U.3 — 打标签步骤(抽取后、核对前)
- 摘要与各 active 主题标签"定义句"做 embedding 相似度匹配(非生成式),超阈值才打,最多 4 个主题标签
- 币种/辖区用受控列表实体匹配;映射不进的进候选队列(T.4)
- 验收:输出标签全部落在 active 闭集或候选队列

## Commit U.4 — 核对 agent(入库前)
- 逐项验证:DOI(resolveDoi 存在 + 解析标题与卡片一致)、URL(可达)、作者/年份(与权威记录交叉)
- 输出**逐字段核对报告** + 状态判定,而非简单通过/拒绝
- 验收:DOI 与标题对不上时该项标 ❌

## Commit U.5 — 状态机落地
- 抽取失败 → `failed`;核对全过 → `pending`(普通)/ `approved`(管理员,沿用现有 isAdmin);有疑点或六要素不全 → `needs_review`(带报告进管理员队列)
- 完整性分两档:硬必填(标题+≥1作者+年份)缺才挡 failed;摘要/直达URL 缺照样入库标 needs_review。**不因缺一项整条丢弃**
- `resourceStatusEnum` 含:uploading/processing/pending/needs_review/approved/rejected/failed
- 验收:差一摘要的文献入 needs_review 而非被拒

## Commit U.6 — 前端上传中心
- `academic-resources.tsx` 三 tab:手填 / DOI·URL / PDF(目前缺 PDF)
- 上传队列 + 进度条 + 百分比 + 当前任务 + 批量进度
- 核对报告在确认弹窗展示每字段 ✅/⚠️/❌
- 验收:三种方式可用,进度与核对结果可视
