# Code 说明 ── 上传管线落库策略修正(两表两状态 + 三入口分流)

> 这是对《Code说明_上传管线U1-U6.md》U.5/U.6 的重要修正,确定 pipeline 中间状态写哪、何时落正式行。
> 核心原则:**正式 `resources` 表只装"用户确认后"的干净数据;抽取/反查/核对的中间过程不写 resources。**

---

## 一、两张表、两套状态,各管各的

**`resources`(正式表)** —— 只在用户点确认后写入。
- 状态枚举 **去掉 `uploading` / `processing`**,只保留审核态:
  `resourceStatusEnum = pending / needs_review / approved / rejected`
- 这些是"够不够格发布"的语义,与"文件处理到哪一步"无关

**`upload_jobs`(临时工作区,新增)** —— 撑批量/PDF 的进度与断点。
```
upload_jobs:
  id              uuid pk
  batchId         uuid            -- 同一次批量上传共享
  ownerId         -> users.id     -- 谁发起的
  kind            enum(pdf | doi_url_batch)
  rawInput        text            -- 文件名 / DOI / URL
  status          enum(queued | processing | extracted | verified | failed)
  extractedData   jsonb           -- 抽取+反查+打标的中间结果(六要素草稿)
  verifyReport    jsonb           -- 逐字段 ✅/⚠️/❌ 报告
  errorMsg        text null
  createdAt / updatedAt
```
- 进度条读这张表;关页面再回来能接着确认;失败/放弃的行烂在这里不影响主数据
- 用户确认后:把 `extractedData` 搬进 `resources`(INSERT 一条 pending/needs_review),该 job 标记完成或删除

> 一句话:`uploading/processing` 是 **upload_jobs** 的状态;`pending/needs_review/approved/rejected` 是 **resources** 的状态。

---

## 二、三入口分流(明确哪个走内存、哪个走 upload_jobs)

| 入口 | 走法 | 落库时机 |
|---|---|---|
| **手填卡片** | 纯内存(无抽取,直接核对) | 确认后 INSERT resources |
| **单条 DOI/URL** | **纯内存**跑完整 pipeline,不碰 upload_jobs | 确认后 INSERT resources |
| **批量 DOI/URL** | 走 **upload_jobs**,逐条处理 | 确认后从 jobs 批量搬入 resources |
| **PDF(单个/多个)** | 走 **upload_jobs**,逐条处理 | 同上 |

设计依据:单条快(几秒出结果)、用户在页面等着,纯内存最轻快,少一次写库;批量/PDF 慢(联网反查+核对可能数分钟)、可能关页面,必须有 upload_jobs 撑进度和断点。

---

## 三、两种流程的具体步骤

### A. 纯内存流程(手填 / 单条 DOI/URL)
1. 后端在内存里跑:抽取 → 反查链接(U.2)→ 打标(retagResources 逻辑)→ 核对(U.4)
2. 把 { 六要素草稿 + verifyReport } 返回前端,**不写任何表**
3. 前端确认弹窗展示逐字段 ✅/⚠️/❌
4. 用户点确认 → 后端按 U.5 判定 status(pending/needs_review)→ INSERT 一条 resources
5. 用户放弃/关页面 → 什么都没落库,无需清理

### B. upload_jobs 流程(批量 / PDF)
1. 前端提交 → 后端为每个文件/DOI 建一条 `upload_jobs`(status=queued,共享 batchId)
2. 后端逐条处理,每步更新该 job:
   - 抽取中 → `processing`(PDF 先抽文字,抽不到才触发占位 OCR)
   - 抽取+反查+打标完成 → `extracted`,中间结果写 `extractedData`
   - 核对完成 → `verified`,报告写 `verifyReport`
   - 任一步失败 → `failed` + `errorMsg`
3. 前端进度条轮询/订阅 `upload_jobs`(按 batchId):百分比 = verified 数 / 总数,显示当前任务、批量进度
4. 全部 verified 后,确认弹窗列出每条的核对报告,用户可逐条勾选/退回
5. 用户点确认 → 后端把选中的 jobs 的 `extractedData` 批量 INSERT 进 resources(各自按 U.5 判 status),对应 jobs 标记完成
6. 未确认的、failed 的 jobs 留在 upload_jobs;可选:加一个定时任务清理 N 天前的废弃 job(只清这张临时表,不碰 resources)

---

## 四、对既有说明的影响

- **U.5 状态机**:`resourceStatusEnum` 删除 `uploading/processing`(它们移到 upload_jobs.status);其余判定规则不变(全 ✅→pending/approved;有 ⚠️/❌ 或六要素不全→needs_review;硬必填缺→该 job failed,不进 resources)
- **U.6 上传中心**:进度条/队列/批量进度的数据源是 `upload_jobs`(按 batchId 聚合),不是 resources
- **测试夹具**:A 组单条可走纯内存路径测;批量把 10 篇一次性提交,走 upload_jobs 测进度+断点+批量确认
- 新增迁移:建 `upload_jobs` 表;改 `resourceStatusEnum`(去掉两个值——注意已有数据迁移,若库里暂无 uploading/processing 行可直接改)

---

## 验收
- 单条 DOI/URL:全程不在 resources 留中间行;放弃后表干净
- 批量 PDF:进度条按 upload_jobs 实时走;中途关页面再进来,能从 extracted/verified 接着确认
- 确认后才出现在 resources,且 status 仅为 pending/needs_review/approved/rejected
- 抽取失败的文件停在 upload_jobs.failed,不污染 resources
