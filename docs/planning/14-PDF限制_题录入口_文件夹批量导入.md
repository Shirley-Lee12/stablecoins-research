# 第一步:PDF 大小限制放开

当前限制 15MB,需要定位并放开。**大小限制通常分三层,必须逐层确认,漏一层改了也没用**:

1. **前端**:上传组件的 `maxSize`/文件校验逻辑
2. **后端**:Express路由的 multer(或等价中间件)`fileSize` 配置
3. **部署平台**:如果经过 Nginx 或 Replit 的某层网关,可能有平台级请求体大小上限,这层**未必能通过应用代码放开**,请 Code 明确检查并汇报是否存在这类硬上限

**目标**:放开到 50MB(如果平台层有更低的硬上限,请如实汇报实际能放开到多少,不要假装已经放开到50MB)。前后端的错误提示文案要保持一致,超过限制时给用户明确的"文件过大,上限XXMb"提示,不能是含糊的上传失败。

**验收**:用一个真实的、20MB以上的PDF实测上传成功。

# 第三步:文件夹批量导入(前端"选择文件夹" + 自动分流 + 人工确认)

## 3.1 前端架构(必读上方的"关键澄清")

- 使用浏览器原生的文件夹选择能力(`<input type="file" webkitdirectory>`,主流浏览器支持;如遇不支持的浏览器,给出"请改用多选文件"的降级提示)
- 用户选择文件夹后,浏览器会把该文件夹(含子文件夹)里的所有文件枚举出来,**这些文件的实际读取和上传,走的是浏览器→服务器的标准HTTP上传**,和现有批量PDF上传是同一套后端机制,不是"服务器读本地路径"
- 前端拿到文件列表后,**按扩展名自动分类**,不上传给用户看之前先做一次分类汇总

## 3.1.1 推荐目录结构(UI 引导,非强制)

**分类靠扩展名识别,不依赖具体目录结构**——用户把文件全堆在一层也能被正确分类。但为了体验更顺、也方便用户自己整理素材,在"选择文件夹"入口旁边展示一份推荐结构说明:

```
materials/
├── reference-lists/
│   ├── 文献清单.md
│   └── 参考文献清单.docx
├── pdfs/                    ← PDF 放这
└── cnki-exports/            ← 知网四种导出文件放这
    ├── xxx.txt              (RefWorks)
    ├── xxx.ent              (EndNote)
    ├── xxx.enw              (NoteExpress)
    └── xxx.es6              (知网研学)
```

- 这份结构说明作为**静态提示文案**展示在文件夹导入界面(比如一个可展开的"推荐目录结构"提示框),不是强制校验规则
- 扫描逻辑本身**递归遍历所有子文件夹**,不管用户是否按推荐结构组织,只要扩展名对得上就能被正确分类(3.2表格);子文件夹路径可以在汇总界面里展示出来(比如"cnki-exports/xxx.txt"这样的相对路径),帮用户确认识别到的文件来源没搞错
- `.doc/.wps` 这两种格式如果用户实际会用到,请补充进"非结构化参考文献列表"这一类的可识别扩展名里(和 `.md/.docx` 走同一条 3.3 的 LLM 拆解子流程),不要求用户必须转成 `.docx`

## 3.2 分类规则

| 扩展名 | 分类 | 处理方式 |
|---|---|---|
| `.pdf` | PDF | 走现有PDF批量上传pipeline |
| `.txt/.ent/.enw/.es6` | 题录 | 走第二步做好的题录导入pipeline |
| `.md/.docx/.doc/.wps` | 非结构化参考文献列表 | 走新增的"LLM拆解+人工确认"子流程(见3.3) |
| 其他 | 忽略 | 汇总里如实告知"已忽略N个不支持的文件",不能静默丢弃不说明 |

## 3.3 非结构化列表(md/docx)的处理子流程——新增能力

这类文件是自然语言写的参考文献列表(不是结构化题录),没法用固定规则可靠拆分,需要走"LLM先拆、人工确认"这条路:

1. 后端抽取文件全文文本
2. 用一次LLM调用,把整段文本拆解成结构化数组:`[{title, authors, year, urlOrDoi(可空)}, ...]`
3. **拆解结果必须先给用户看、人工确认无误,才能继续**——这是项目"AI解析内容必须人工确认才能进入下一步"这条铁律的延伸,不能因为是新功能就绕过。前端展示一个可编辑的表格(每行:标题/作者/年份/链接),用户可以修正拆错的行、合并/删除多余的行
4. 确认后,逐行分流:
   - 有 `urlOrDoi` → 走现有DOI/URL批量导入pipeline(复用,不新写)
   - 没有 → 走U.2标题反查pipeline(复用现有 `resolveLink()` 的标题搜索路径,不新写)
5. 后续和其他入口一样,进 `upload_jobs`,不绕过任何一层核对/审核

## 3.4 汇总确认界面

- 分类完成后(3.2),上传**之前**,先给用户看一份汇总:"识别到 X 个PDF、Y 个题录文件、Z 份参考文献列表(共解析出W条待人工确认)、已忽略N个不支持的文件",允许用户在这一步取消勾选不想处理的文件
- 确认后,才把各分类批次分别送进各自pipeline,统一走 `batchId` 分组的进度显示(复用04号文档设计的续传能力,folder scan产生的所有job应该能共享一个更高层的"这次文件夹导入"分组标识,方便用户看到"这次从文件夹导入的进度",而不只是零散的各个batchId)
- **人工确认再入库**这条原则不变——文件夹扫描只是让"抽取+分流+跑pipeline"这部分自动化、批量化,最终是否落库,仍然由用户在确认界面逐条/批量确认(全部核对无误的可以批量一键确认,有问题的单独处理)
